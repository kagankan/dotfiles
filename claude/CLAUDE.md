# CLAUDE.md (user-level)

## 出力ルール

### 調査結果・レビュー結果

- `.ai-output/sessions/` 配下にセッションごとのディレクトリを作り、markdown で書き出す
- ディレクトリ名は `${作成日}-${CLAUDE_CODE_SESSION_ID}-${日本語で概要}` とする
  - 作成日は `date "+%Y-%m-%d"` で取得
  - 概要は作業内容を短く表したもの（例: `2026-06-13-abc123-PR レビュー`）
- セッション内で2回目以降の書き出しでは、既存ディレクトリを探して再利用する:
  ```sh
  find .ai-output/sessions -maxdepth 1 -name "*${CLAUDE_CODE_SESSION_ID}*" -type d
  ```
  - 見つかればそのディレクトリを使う。なければ新規作成する
- ファイル名は `${日時}-${日本語タイトル}.md` とする
  - 日時は `date "+%Y-%m-%d-%H-%M"` で取得

### コマンド実行の出力

- `pnpm test` / `pnpm lint` 等の長時間コマンドを実行するときは、出力を必ずファイルに保存する
- 出力先はセッションディレクトリ配下の `logs/` を使う（`/tmp/` は permission の都合で避ける）
  - パス: `.ai-output/sessions/${セッションディレクトリ}/logs/`
  - `.ai-output/` はグローバル gitignore に登録済み
- ファイル名は `$(date +%Y-%m-%d-%H-%M)-<内容>.log` の形式（例: `2026-04-29-02-15-test.log`）
- 結果の確認は保存したファイルを `grep` / `sed` で処理する。同じコマンドを再実行しない
- 例:
  - 実行: `pnpm test --run path/to/file 2>&1 | tee ${SESSION_DIR}/logs/$(date +%Y-%m-%d-%H-%M)-test.log`
  - 確認: `grep -E 'Test Files|Tests |✗|FAIL' ${SESSION_DIR}/logs/2026-04-29-02-15-test.log`
  - 別観点で見直したいときは grep パターンを変えるだけで再実行不要

## Pull Request のルール

- PR は必ず **Draft** 状態で作成する。Open に切り替えるのは人間が行う
  - `gh pr create` を使う場合は `--draft` フラグを必ず付ける
  - 既存 PR を作り直す場合も Draft で作成する
- PR description は、リポジトリに `.github/pull_request_template.md` がある場合は**そのテンプレートのセクション構成をそのまま使って**記入する
  - `## Summary` / `## Test plan` 等の独自構成に書き換えない
  - テンプレートのコメント (`<!-- ... -->`) は埋めるかコメントごと差し替える
  - 既存のチェックリスト項目は AI 側で完了確認したものだけ `[x]`、人間が後でやる項目（セルフレビュー、AI レビュー、ラベル付与、Slack 連絡等）は `[ ]` のまま残す
  - テンプレートが無いリポジトリのみ Summary / Test plan で書く
- 書く必要のない項目は削除し、常体で簡潔に書く

## 技術的な選択肢を検討・提示するときのルール

- 複数の技術的選択肢を検討するときは、すぐに結論へ飛ばず、まず評価の観点（軸）を広げて洗い出してから比較する
- ユーザーに選択を委ねるときは、各選択肢の**メリット・デメリットを必ず併記する**
- 補足情報として、どの選択肢が一般的か（著名な OSS や有名プロジェクトでの採用例など）が分かれば添える

## 日本語の表記ルール

日本語を囲む括弧は全角を使用する（例: `（超過時に〜と表示）`）。コード中のコメントや tsdoc にも適用する。

## lint ルールの追加・アップデート時に出現した新たなエラーへの対応方針

- コード動作に影響を与えるような変更は行わない。つまり、コメントの追加・削除のみ行う
- そのブランチ内での対応が難しいものは、該当行に `eslint-disable` で打ち消す（ルールを off にはしない）
